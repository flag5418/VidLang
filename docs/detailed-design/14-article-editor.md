# VidLang 文章编辑器 — 批量导入 + Markdown 编辑器设计

> 核心：两种创建方式（批量导入 / Markdown 编辑器），编辑器支持图片插入。

---

## 一、两种创建文章的入口

```
ArticleListPage（文章列表）
├── [+ New] 按钮
│   ├── 📥 Batch Import      ← 批量导入
│   └── ✏️ New Article       ← 新建文章（打开编辑器）
│
└── [⋮] 菜单（文章卡片上）
    └── Edit → 打开该文章的 Markdown 编辑器
```

### 2.3 批量导入的处理流程

```

    ↓

├── 读取内容为字符串
├── 文件名作为文章标题
├── 检测是否为 .md（根据扩展名）

    ↓

├── Successfully imported: 3 articles
├── Failed: 0
└── [View Articles] → 跳转到文章列表
```

### 3.2 工具栏说明

| 按钮 | 操作 | 插入的 Markdown |
|------|------|----------------|
| **B** | 加粗 | `**text**` |
| *I* | 斜体 | `*text*` |
| **H1** | 一级标题 | `# text` |
| **H2** | 二级标题 | `## text` |
| **Q** | 引用 | `> text` |
| **L** | 列表 | `- text` |
| **🖼** | 插入图片 | `![](filename.jpg)` |
| **📷** | OCR 识别 | 拍照 → 识别文字 → 插入到光标位置 |

**关于 Markdown 语法对普通用户的门槛：**

用户不需要懂 Markdown 语法。编辑器提供两种模式：

```dart
// 模式 1：纯文本模式（默认）
// 用户只看到普通文本
// 点 [B] 按钮 → 选中的文字变粗（编辑器用富文本显示）
// 底层自动转为 Markdown 存储

// 模式 2：源码模式（可选）
// 用户看到 Markdown 标记
// 适合懂 Markdown 的用户
```

**实现建议：** 第一次做可以就用**纯文本编辑区**（TextField），用户粘贴内容自动保留段落。工具栏插入 Markdown 标记。用户不需要理解 `#` 和 `**` 的含义，他们只需要知道"点 H1 按钮就是大标题"。

### 3.3 图片插入的处理

```
用户点击 [🖼] 按钮
    ↓
弹出选择方式：
├── 📷 Take Photo（拍照）
├── 🖼 Choose from Gallery（相册）
└── 📁 Choose from Files（文件选择器）
    ↓
选择图片后
    ↓
系统自动处理：
├── 生成唯一文件名：img_{uuid}.jpg
├── 复制图片到 Article 的专属目录：
│   {appCacheDir}/articles/{articleCode}/images/
├── 在编辑区光标位置插入：
│   ![描述文字](img_{uuid}.jpg)
└── 如果用户没有输入描述文字，用空描述
    ↓
编辑区显示为占位符或缩略图：
┌──────────────────────────────┐
│  ![chart](img_001.jpg)      │ ← Markdown 语法（源码模式）
│                              │
│  ┌──────────────────┐       │ ← 或者显示图片缩略图
│  │   (image)        │       │
│  └──────────────────┘       │
└──────────────────────────────┘
```

### 3.4 图片存储方案

```
文章图片存储路径：
{appCacheDir}/articles/{articleCode}/images/
├── img_001.jpg
├── img_002.jpg
└── img_003.jpg

文章 Markdown 中引用：
![chart](img_001.jpg)
![Source: NASA](img_002.jpg)

阅读器中加载图片：
// flutter_markdown 的 ImageBuilder
MarkdownBody(
  data: articleContent,
  imageBuilder: (uri, title, alt) {
    final imagePath = '${articleImageDir}/${uri.path}';
    return Image.file(File(imagePath));
  },
)
```

**关于导出/同步时图片的处理：**

| 场景 | 处理方式 |
|------|---------|
| 本地阅读 | 直接从本地目录加载 ✅ |
| WiFi 传输 | 不需要将图片传回电脑（阅读在手机上） |
| 云同步（后续） | 将图片上传到 Supabase Storage，更新 Markdown 中的路径 |

### 3.5 OCR 识别 + 图片插入的协同

```
用户拍照后，有两个选项：
┌──────────────────────────────────────┐
│  Photo captured                      │
├──────────────────────────────────────┤
│                                       │
│  📝 Recognize Text (OCR)             │ ← 推荐
│  Extract text and insert into editor │
│                                       │
│  🖼 Insert as Image                   │
│  Add the photo to the article        │
│                                       │
│  [Retake]                             │
└──────────────────────────────────────┘
```

这样用户一次拍照，既可以识别文字，也可以保留图片。甚至两个都做：

```
拍照 → OCR 识别文字 → 文字插入编辑器
     → 同时将图片保存 → 在 OCR 文字下方插入 ![](img.jpg)
```

## 六、关于"图片好不好处理"的回答

### 6.1 技术可行性

| 环节 | 难度 | 方案 |
|------|------|------|
| 拍照获取图片 | ⭐ 简单 | `ImagePicker` / 相机 channel |
| 保存图片到本地 | ⭐ 简单 | `File.copy()` 到缓存目录 |
| Markdown 中引用图片 | ⭐ 简单 | `![](path)` 语法 |
| 阅读器加载图片 | ⭐⭐ 中等 | `flutter_markdown` 的 `imageBuilder` 回调 |
| 图片管理（删除/替换） | ⭐⭐ 中等 | 编辑器显示已插入的图片列表 |
| 云同步图片（后续） | ⭐⭐⭐ | Supabase Storage + 路径替换 |

**结论：图片插入完全可行，开发成本不高。**

### 6.2 建议的限制

```
建议初期加一些限制，降低复杂度：

1. 图片格式：只支持 JPEG / PNG / WebP
2. 单篇文章最大图片数：20 张
3. 单张图片最大尺寸：10MB（超过则压缩）
4. 图片自动命名：img_001.jpg, img_002.jpg
5. 删除文章时：自动清理所有关联图片
```

### 6.3 图片使用场景的价值判断

```
对有图片的文章的价值：

✅ 数据图表/曲线图 → 高价值
   用户学经济学/科学类文章时，图表是核心内容
   OCR 无法准确识别图表数据

✅ 带有插图的文章 → 中等价值
   插图帮助理解文章内容

✅ 地图/示意图 → 高价值
   地理/历史类文章

✅ OCR 效果不好时的后备方案 → 高价值
   拍糊了、字体特殊 → 图片比错误文字有用

❌ 纯文字页面的照片 → 低价值（应该用OCR）
   直接识别成文字更好

结论：值得做，而且实现了"不管什么来源都能导入"的承诺
```

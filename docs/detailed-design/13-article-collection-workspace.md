# VidLang 文章收集工作台 — 碎片拼接设计

> 核心思路：用户不是"导入一篇文章"，而是"收集碎片、拼成文章"。

---

## 一、设计理念

### 用户真实行为分析

```
用户想学一篇外刊文章 →
├── 在 Safari 打开 → 复制一段
├── 又看到另一段相关 → 再复制一段
├── 翻到一张数据图表 → 截屏
├── 想到自己的笔记 → 手动打几行字
└── 最终：这些碎片要拼成一篇完整的文章
```

**传统的"导入文件"模式不适合这种场景。**

### 核心理念：Article Collection Workspace（文章收集工作台）

```
不是一次性导入，而是反复收集 → 最终合成
```

```
打开工作台
    ↓
┌──────────────────────────────────────┐
│  收集阶段：反复操作，内容不断追加     │
│                                      │
│  📋 从浏览器复制          → 粘贴追加 │
│  📷 拍一页书              → OCR追加 │
│  ✏️ 手动写一段感想        → 文本追加 │
│  📋 再从 PDF 复制一段     → 粘贴追加 │
│  📷 再拍一页              → OCR追加 │
│                                      │
│  ↓ 收集完毕                               │
│  [Process & Create]                     │
│  ↓                                       │
│  统一解析 → 分章 → 分句 → 生成时间轴      │
│  → 进入阅读器                            │
└──────────────────────────────────────┘
```

### 2.2 追加操作细节

#### 📋 Paste（粘贴）

```
用户点击 [📋 Paste]
    ↓
检测剪贴板内容
├── 有文本 → 自动读取，弹出确认
│   └── "Paste "Climate change is..."? [Cancel] [Paste]"
│
├── 剪贴板为空 → 显示提示
│   └── "Copy text first, then tap Paste"
│
└── 有图片 → 暂不支持（未来可 OCR）
    ↓
确认后 → 内容追加为新的 Fragment
    ↓
自动滚动到最新 Fragment
```

#### 📷 Scan（拍照 OCR）

```
用户点击 [📷 Scan]
    ↓
调用 IosNativeFeatures.extractTextFromCamera()
    ↓
OCR 识别完成 → 显示识别结果
┌──────────────────────────────────────┐
│  OCR Result                          │
├──────────────────────────────────────┤
│                                       │
│  Scientists warn that without        │

│  could rise...                       │
│                                       │
│  [Retake]  [Edit]  [Add to Article]  │
└──────────────────────────────────────┘
    ↓
确认后 → 内容追加为 Fragment（来源标记为 📷）
```

#### ✏️ Type（手动输入）

```
用户点击 [✏️ Type]
    ↓
弹出键盘输入框
┌──────────────────────────────────────┐
│  Type or paste text                  │
├──────────────────────────────────────┤
│                                       │
│  (用户手动输入内容)                   │
│                                       │
│  [Cancel]  [Add]                     │
└──────────────────────────────────────┘
    ↓
确认后 → 内容追加为 Fragment（来源标记为 ✏️）
```

### 2.3 Fragment 的操作

#### 长按排序

```
用户长按某个 Fragment
    ↓
进入拖拽模式（HapticFeedback）
    ↓
上下拖动 → 重新排序 Fragment 顺序
    ↓
松手 → 确认新位置
```

#### 左滑删除

```
用户在 Fragment 上左滑
    ↓
显示红色 [Delete] 按钮
    ↓
点击删除 → Fragment 移除
```

#### 点击编辑

```
用户点击 Fragment 的 [✏️] 按钮
    ↓
弹出全屏文本编辑器（可滚动）
┌──────────────────────────────────────┐
│  Edit Fragment                       │
├──────────────────────────────────────┤
│                                       │
│  （完整的可编辑文本区）               │
│                                       │
│  [Cancel]  [Save]                    │
└──────────────────────────────────────┘
    ↓
保存 → 更新 Fragment 内容
```

## 三、电脑端设计（WiFi 页面）

### 3.1 设计原则

```
电脑端 ≠ App 端的功能复制
电脑端更简单，因为：
1. 电脑有键盘 → 粘贴比手机方便
2. 电脑没有相机 → 不做 OCR
3. 电脑屏幕大 → 一个文本框就够了
```

### 3.2 WiFi 页面文章收集区

```
┌────────────────────────────────────────────────────────┐
│  📖 Create New Article                                 │
│                                                         │
│  ┌────────────────────────────────────────────────┐    │
│  │  Paste your article text here...               │    │
│  │                                                 │    │
│  │  You can:                                       │    │
│  │  • Paste from browser (Ctrl+V)                  │    │
│  │  • Paste from PDF/Word (Ctrl+V)                 │    │
│  │  • Type directly                                │    │
│  │  • Upload .md or .txt file                      │    │
│  │                                                 │    │
│  │  ──────────────────────────────────────         │    │
│  │                                                 │    │
│  │  Climate change is one of the most pressing     │    │ ← 用户粘贴的完整内容
│  │  issues of our time...                          │    │
│  │                                                 │    │
│  │  ## Rising Temperatures                         │    │
│  │                                                 │    │
│  │  Global temperatures have risen by an            │    │
│  │  average of 1.2°C...                            │    │
│  │                                                 │    │
│  └────────────────────────────────────────────────┘    │
│                                                         │
│  Characters: 1,284  |  [Clear]  [Upload .md/.txt]     │
│                                                         │
│  ┌────────────────────────────────────────────────┐    │
│  │           [Create Article]                     │    │
│  └────────────────────────────────────────────────┘    │
└────────────────────────────────────────────────────────┘
```

**为什么电脑端不用"Fragment 列表"？**
- 电脑端用户习惯一次性粘贴（Ctrl+A → Ctrl+C → Ctrl+V）
- 电脑键盘操作比手机快得多，不需要逐段追加
- 如果用户真的需要多次粘贴，他们可以在文本框内任意位置粘贴
- 保持极简，降低开发成本

### 3.3 电脑端文本处理流程

```
用户粘贴多段内容（可能包含 HTML 格式文本）
    ↓
系统自动清理格式
├── 移除多余空行（保留段落间距）
├── 保留基本结构（段落/空行/列表）
├── 检测是否包含 Markdown 标记
└── 统一编码（UTF-8）
    ↓
显示在文本区（用户可手动编辑调整）
    ↓
用户点击 [Create Article]
    ↓
统一处理管线
├── 清洗文本
├── 提取标题（首行或用户输入）
├── 分章分句
└── 保存 → 进入阅读器
```

### 3.4 WiFi 页面"追加粘贴"的备选方案

如果用户确实需要多次粘贴，可以在文本框下方增加一个辅助按钮：

```
┌────────────────────────────────────────────────────────┐
│  Text Content                                          │
│                                                         │
│  ┌────────────────────────────────────────────────┐    │
│  │  (main text area)                               │    │
│  │                                                 │    │
│  │  Previously pasted content...                   │    │
│  │                                                 │    │
│  └────────────────────────────────────────────────┘    │
│                                                         │
│  [📋 Append from Clipboard]  ← 点击后自动读取剪贴板     │
│                             追加到文本框末尾            │
│                                                         │
│  Characters: 1,284  |  [Clear]  [Upload File]          │
│                                                         │
│  [Create Article]                                       │
└────────────────────────────────────────────────────────┘
```

用户操作流程：
1. 从浏览器复制第一段 → 粘贴到文本框
2. 切回浏览器复制第二段 → 切回 WiFi 页面
3. 点击 [📋 Append from Clipboard] → 自动追加
4. 重复直到收集完毕
5. 点击 [Create Article]

## 四、多张截图拼接的处理

### 4.1 手机端多截图场景

```
用户拍了一本书的 5 页
    ↓
第1次拍照：[📷 Scan] → OCR → Fragment 1（📷 Page 1）
第2次拍照：[📷 Scan] → OCR → Fragment 2（📷 Page 2）
第3次拍照：[📷 Scan] → OCR → Fragment 3（📷 Page 3）
    ...
    ↓
5 个 Fragment 显示在工作台中
    ↓
用户可以：
├── 检查每个 Fragment 的文字是否正确
├── 拖动排序（如果拍乱了顺序）
├── 删除拍糊的 Fragment
└── 编辑 OCR 识别错误的文字
    ↓
[Process & Create] → 5个 Fragment 拼接 → 统一分章分句
```

### 4.2 拼接逻辑

```dart
// 用户点击 [Process & Create] 时
// 将所有 Fragment 的内容按顺序拼接

final fullText = fragments
    .where((f) => f.text.isNotEmpty)
    .map((f) => f.text.trim())
    .join('\n\n');  // 每个 Fragment 之间用双换行分隔

// 然后统一走处理管线：
// fullText → 清洗 → 分章 → 分句 → 保存
```

## 五、电脑端和手机端的能力对比

| 能力 | 手机端 (App) | 电脑端 (WiFi) |
|------|-------------|---------------|
| 粘贴文本 | ✅ App 内粘贴 | ✅ 浏览器粘贴 |
| 多次追加粘贴 | ✅ Fragment 列表 | ✅ Append 按钮 |
| 手动输入 | ✅ 弹出键盘 | ✅ 直接打字 |
| OCR 拍照 | ✅ iOS Vision | ❌ 无摄像头 |
| 文件导入 | ✅ .md / .txt | ✅ .md / .txt |
| 拖拽排序 | ✅ 长按拖动 | ❌ (文本框内编辑) |
| 编辑片段 | ✅ 弹窗编辑 | ✅ 文本框直接编辑 |
| 删除片段 | ✅ 左滑删除 | ✅ 文本框内删除 |
| 标题编辑 | ✅ 独立输入框 | ✅ 独立输入框 |

## 六、关键设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 电脑端用文本框还是 Fragment 列表？ | **文本框** | 电脑用户习惯一次性粘贴，Fragment 列表过度设计 |
| 手机端用 Fragment 列表还是文本框？ | **Fragment 列表** | 手机用户需要多次追加/拍照/排序，Fragment 更直观 |
| 是否支持在 Fragment 中间插入新内容？ | **仅追加到末尾** | 用户可以通过排序调整顺序，插入太复杂 |
| 多张截图识别出错怎么办？ | **Fragment 可编辑** | 用户可以在工作台中修正 OCR 错误 |
| 文章标题怎么生成？ | **从内容提取**（首行/# 标题/前 50 字） | 减少用户操作步骤 |
| 用户点"创建"后还能修改吗？ | **可以在阅读器内重新编辑文章** | 降低用户"一次做对"的焦虑 |
